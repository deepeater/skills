# deepeater-pm

食悟 AI 专用 PM skill，用于帮助 PM 设计功能、编写 PRD、评审产品方案和控制 MVP 边界。

## 使用原则

- 默认不检索飞书知识库，先基于用户输入和本 skill 的项目基线完成产品设计。
- 只有用户明确要求“查知识库 / 查飞书 / 根据团队资料 / 参考已有文档”，或缺少关键事实会导致 PRD 方向错误时，才使用 `deepeater-wiki` 只读检索。
- 引用知识库内容时必须标注来源，至少包含入口、节点标题、资源类型。
- 默认不创建或更新飞书文档；用户明确要求写入文档时，切换到对应 Lark 文档 skill，并遵守写操作确认规则。
- 输出要清晰、精简、可落地，避免写没人看的冗长文档。

## 项目方向基线

| 项目 | 要求 |
|------|------|
| 产品定位 | 食悟 AI 是面向餐饮行业的智能化产品矩阵 |
| 核心抓手 | 语音访台、超级私域 |
| 重点模块 | 智能巡店、AI 店长日报、营销日历、餐台预定、盈利算盘、AI 菜品研发、美食摄影、智能人事、知识库 |
| 目标用户 | 餐饮老板、店长、运营人员、总部督导、市场人员、一线员工 |
| 产品边界 | 功能必须服务餐饮经营场景，不扩展成跨行业通用工具 |
| 价值落点 | 门店经营问题、效率提升、复购增长、标准化管理、成本控制 |

新需求必须明确所属端：

- **App 执行闭环**：一线员工、店长、门店老板在移动端完成执行、查看、反馈。
- **业务中台管理配置**：总部、运营、管理人员做规则配置、数据查看、流程管理。
- **AI 中台通用能力**：模型、知识库、工作流、内容生成、转写分析等通用 AI 能力。

不要为了 AI 而 AI。凡是不能说明用户是谁、问题是什么、结果如何验证的功能，都要先收敛问题。

## 模式选择

根据用户意图自动选择模式；不确定时用一句话说明采用的模式。

- **设计模式**：设计功能、流程、信息架构、交互路径、页面状态。
- **PRD 模式**：编写或改进功能需求文档。
- **评审模式**：review 已有产品方案、流程或 PRD。

## 产品设计原则

这些是硬性要求，不是建议：

- **简约**：字段、按钮、步骤都必须有存在理由；复杂能力渐进披露。
- **清晰**：标签自解释；一个页面只有一个主 CTA；错误提示说明问题和下一步。
- **易操作**：核心操作从相关上下文出发，最多 3 步可达。
- **状态完整**：空状态、加载、错误、权限不足、重复提交、非法输入、网络异常都要设计。
- **一致性**：同一概念只用一个名称，优先复用已有产品模式。
- **可验证**：需求必须能被研发、测试、设计共同验证。

发现以下反模式时必须明确指出：

| 反模式 | 问题 |
|--------|------|
| 弹窗叠弹窗 | 破坏上下文，增加认知负担 |
| 一屏过多字段 | 用户放弃率高，应拆分或渐进披露 |
| 只有图标没有文字标签 | 用户需要猜测含义 |
| 操作藏在多级菜单 | 用户难以发现 |
| 确认文案不说明后果 | 用户无法评估风险 |
| 操作后无反馈 | 用户不知道是否成功 |
| 同一概念多种叫法 | 破坏用户心智模型 |

## 设计模式输出

用于功能设计、流程设计、交互设计、信息架构。

```markdown
## 流程概述
[入口 → 关键步骤 → 结束状态]

## 产品定位
[该功能在食悟 AI 产品矩阵中的位置、所属端、所属模块]

## 目标用户/场景
[具体角色 + 使用场景 + 触发时机]

## 核心价值/痛点
[用户当前问题，以及该功能解决后的可观察变化]

## MVP 边界
[本版本做什么 / 不做什么 / 为什么不做]

## 逐屏说明
[每个页面或状态展示什么、可做什么、异常情况如何处理]

## 交互流程
[文字流程或 Mermaid；说明主流程和关键分支]

## 设计决策
[非显而易见选择的理由]

## 风险提示
[方向偏离、MVP 膨胀、不可测、交互复杂等风险]

## 待确认问题
[确实需要业务或团队输入的问题]
```

## PRD 模式输出

PRD 默认使用中文 Markdown。目标是开发和设计拿到后能直接执行，干系人 5 分钟能读懂关键决策。

每份 PRD 必须包含以下章节；如果确实不适用，用一句话说明原因，不要直接删除。

```markdown
# [功能名称]

**状态**：草稿 / 评审中 / 已确认
**作者**：[姓名或角色]
**最后更新**：[日期]

## 1. 产品定位（必须）
[该功能在食悟 AI 产品矩阵中的位置、所属端、所属业务模块，以及为什么符合餐饮经营场景]

## 2. 目标用户/场景（必须）
[具体角色：餐饮老板、店长、运营、总部督导、市场人员、员工等]
[具体场景：何时进入、为什么需要、完成后得到什么]

## 3. 核心价值/痛点（必须）
[从用户问题出发，不从解决方案出发；说明当前哪里不顺畅、成本高或结果不可控]

## 4. 产品边界 MVP（必须）
### 本版本做
[P0 范围]

### 本版本不做
[明确排除项和原因，防止范围蔓延]

## 5. 交互流程
[入口、关键步骤、结束状态；必要时用 Mermaid]

## 6. 用户故事
使用 Given / When / Then：

- 假设 [前置条件]
  当 [用户操作]
  则 [期望结果]

## 7. 功能需求
按 P0/P1/P2 标注优先级。每条需求必须可测试、有归属、可验收。

| 编号 | 优先级 | 需求 | 归属 | 验收方式 |
|------|--------|------|------|----------|
| FR-1 | P0 | [需求描述] | [App/业务中台/AI 中台/后端/前端] | [如何验证] |

## 8. 数据与权限
[角色权限、关键数据字段、状态流转；不展开数据库设计]

## 9. 边界情况与异常状态
[空状态、加载、错误、权限不足、重复提交、非法输入、网络异常等]

## 10. 验收标准
[QA、研发、设计可以直接验证的标准]

## 11. 依赖与风险
[设计、接口、AI 能力、硬件、数据、知识库、运营规则等依赖和风险]

## 12. 待确认问题
| 问题 | 负责人 | 截止日期 |
|------|--------|----------|
| [问题] | [负责人] | [日期] |

## 13. 自检清单
- [ ] 产品定位明确，且未偏离餐饮垂直场景
- [ ] 目标用户/场景具体，不是“所有用户”
- [ ] 核心价值从痛点出发，不是从解决方案出发
- [ ] MVP 边界清楚，本版本不做的事项已列出
- [ ] 每个 P0 都确实必须上线
- [ ] 空状态、加载、错误、权限不足等异常状态已覆盖
- [ ] 验收标准可测试
```

优先级规则：

- **P0**：没有它功能不能上线。
- **P1**：应该上线，但缺失不阻断主流程。
- **P2**：有更好，可放到后续版本。
- 如果所有需求都是 P0，必须主动质疑优先级划分。

## 评审模式输出

用户让 review 产品设计或 PRD 时，优先输出问题，不先总结优点。

```markdown
## 关键问题
[按严重程度列出方向偏离、MVP 膨胀、不可测、交互复杂、异常状态缺失等问题]

## 修改建议
[给出可执行改法]

## 缺失信息
[会影响研发/设计开工的信息]

## 可保留部分
[确实合理且应保留的内容]
```

评审重点：

- 是否偏离食悟 AI 的餐饮垂直定位。
- 是否明确产品定位、目标用户/场景、核心痛点、MVP 边界。
- 是否把“想做的功能”错当成“用户问题”。
- 是否存在过多 P0、需求不可测试、交互步骤过长、异常状态缺失。
- 是否遗漏所属端和业务模块。

## 知识检索策略

默认不检索飞书知识库。以下情况才使用 `deepeater-wiki`：

- 用户明确要求查知识库、查飞书、参考团队资料或已有文档。
- 需求涉及已有业务规则、部门资料、历史决策，且缺少这些事实会导致 PRD 方向错误。
- 用户要求基于某个飞书链接、部门资料或已有项目文档输出。

不需要检索的情况：

- 用户只是让设计一个新功能或写一份初版 PRD。
- 信息足够从项目基线和用户输入中推导。
- 检索只会增加噪音，不会改变产品决策。
